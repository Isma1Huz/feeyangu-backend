# Payment Architecture

FeeYangu uses a multi-tenant (per-school) payment architecture supporting M-Pesa
PayBill (C2B), M-Pesa STK Push, KCB Bank, Equity Bank, NCBA, and Co-operative Bank.

---

## Key Tables

| Table | Purpose |
|-------|---------|
| `payment_transactions` | One row per individual payment event; status: pending → processing → completed/failed |
| `student_payment_accounts` | One stable BillRefNumber per student per school (reused across all payments) |
| `receipts` | One receipt per completed PaymentTransaction |
| `invoice_payments` | Maps completed transactions to invoices; supports partial allocation |
| `invoices` | Per-student invoice; `paid_amount` / `balance` updated on each completion |
| `school_api_credentials` | Per-school API keys/secrets for each provider (encrypted) |
| `school_payment_configs` | Per-school PayBill shortcodes, account names, sort order |
| `payment_webhook_logs` | Raw webhook log for every inbound callback |

---

## Stable Payment Reference (BillRefNumber)

Every student gets a single permanent reference stored in `student_payment_accounts.reference`.

Format: `{NORMALIZED_ADMISSION}-{SCHOOL_ID}-{RANDOM_CODE}` (max 30 chars, A-Z 0-9 and `-` only)

Example: `995-5-DKAKJD`

Generated by `App\Services\Payment\PaymentReferenceGenerator`.

This reference is:
- Used as the M-Pesa **AccountReference** in STK Push requests
- Used as the M-Pesa **BillRefNumber** for manual PayBill payments
- Included in bank payment callback references/narrations

## Payment Flows

### 1. M-Pesa STK Push

```
Parent initiates → ParentPaymentController::initiate()
  └─ getOrCreatePaymentAccount(student) → StudentPaymentAccount.reference
  └─ PaymentTransaction::create(status=pending, reference=account.reference)
  └─ MpesaPaymentProvider::initiatePayment()
       └─ OAuth token from Daraja (per-school credentials, cached 3500s)
       └─ POST /mpesa/stkpush/v1/processrequest
            AccountReference = account.reference
            CallBackURL = /api/payments/callback/mpesa/{school_id}
       └─ transaction.status = 'processing'
       └─ transaction.provider_reference = CheckoutRequestID

Safaricom calls back → POST /api/payments/callback/mpesa/{school_id}
  └─ PaymentCallbackController::handle()
  └─ MpesaPaymentProvider::handleCallback()
  └─ handleStkCallback()
       └─ Lookup transaction by provider_reference=CheckoutRequestID
       └─ completeTransaction() [DB transaction, lock-for-update]
            └─ transaction.status = 'completed'
            └─ transaction.provider_reference = MpesaReceiptNumber
            └─ Receipt::firstOrCreate()          ← idempotent
            └─ InvoiceAllocationService::allocate()
```

### 2. M-Pesa PayBill (C2B / Manual)

```
Customer pays via PayBill shortcode using BillRefNumber = account.reference

Safaricom calls back → POST /api/payments/callback/mpesa/{school_id}
  └─ MpesaPaymentProvider::handleC2BCallback()
       └─ Idempotency check: PaymentTransaction.where(provider=mpesa, provider_reference=TransID)
       └─ Lookup StudentPaymentAccount by reference=BillRefNumber
       └─ [DB transaction]
            └─ PaymentTransaction::create(status=completed, amount=TransAmount, provider_reference=TransID)
            └─ Receipt::create()
            └─ InvoiceAllocationService::allocate()
```

Each C2B payment creates a **new** `PaymentTransaction` row, so partial payments are
supported: a student can pay KES 2,000 today and KES 3,000 next week using the same
BillRefNumber.

### 3. Bank Providers (KCB, Equity, NCBA, Co-op)

```
Parent initiates → ParentPaymentController::initiate()
  └─ PaymentTransaction::create(status=pending)
  └─ BankProvider::initiatePayment()
       └─ Load credentials from SchoolApiCredential
       └─ POST to bank API
       └─ transaction.status = 'processing'

Bank calls back → POST /api/payments/callback/{provider}/{school_id}
  └─ BankProvider::handleCallback()
  └─ markAsCompleted()
       └─ Receipt::firstOrCreate()
       └─ InvoiceAllocationService::allocate()
```

## Callback URLs

All callback URLs include `{school_id}` so they work independently of subdomain routing.

| Provider | Callback URL |
|----------|-------------|
| M-Pesa   | `POST /api/payments/callback/mpesa/{school_id}` |
| KCB      | `POST /api/payments/callback/kcb/{school_id}` |
| Equity   | `POST /api/payments/callback/equity/{school_id}` |
| NCBA     | `POST /api/payments/callback/ncba/{school_id}` |
| Co-op    | `POST /api/payments/callback/cooperative/{school_id}` |

## Invoice Allocation

`App\Services\Payment\InvoiceAllocationService::allocate(PaymentTransaction $tx)`:

1. Finds all open invoices (`status` in `sent`, `partial`, `overdue`) for the student,
   ordered by `due_date ASC` (oldest first / FIFO).
2. Applies the payment amount until exhausted.
3. Creates `InvoicePayment` records (one per invoice touched).
4. Updates `invoice.paid_amount`, `invoice.balance`, and `invoice.status`.

The service is **idempotent**: it sums existing `InvoicePayment.amount_applied` for
the transaction before applying any new allocation, so re-running it is safe.

The parent dashboard `paid_amount` / `balance` displayed for each child derives
directly from `invoices.paid_amount` and `invoices.balance`, which are updated
synchronously within the callback DB transaction. Balance therefore reflects the
payment immediately after the callback is processed.

## Idempotency

| Scenario | Guard |
|----------|-------|
| STK Push callback received twice | `PaymentTransaction.status = completed` check + `Receipt::firstOrCreate` |
| C2B callback received twice | `unique(provider, provider_reference)` DB constraint on `payment_transactions` |
| Invoice allocation run twice | Pre-existing `InvoicePayment` rows reduce remaining amount to 0 |
| Bank callback received twice | `Receipt::firstOrCreate` |

## Per-School Credentials

Stored in `school_api_credentials` (encrypted JSON blob via `encrypted:array` cast).

| Field | Description |
|-------|-------------|
| `school_id` | Foreign key to schools |
| `provider` | `mpesa`, `kcb`, `equity`, `ncba`, `coop` |
| `environment` | `sandbox` or `production` |
| `enabled` | Boolean |
| `credentials` | Encrypted JSON: provider-specific keys (see below) |

### M-Pesa credential keys
- `consumer_key`, `consumer_secret`, `passkey`, `shortcode`

### KCB credential keys
- `api_key`, `api_secret`, `account_number`, `base_url` (optional)

### Equity credential keys
- `api_key`, `merchant_code`, `base_url` (optional)

### NCBA / Co-op credential keys
- `api_key`, `account_number`, `base_url` (optional)

## Partial Payments

The `payment_transactions.reference` column no longer has a unique constraint.
Multiple `PaymentTransaction` rows can share the same `reference` (the student's stable
BillRefNumber). The unique constraint that was removed is replaced by:

- `unique(provider, provider_reference)` — prevents duplicate processing of the same
  provider transaction ID (M-Pesa TransID / bank transaction reference).
